# HyperImmunISE

HyperImmunISE is a computational framework for designing glycan-mediated masking strategies to modulate protein immunogenicity.  
The platform integrates structural modelling, accessibility analysis, and rule-based site selection to support rational construct design.

It is intended for researchers in antibody engineering, vaccine design, and computational immunology who require reproducible, automatable workflows.

Full documentation, advanced configuration, and dependency details are available at:  
https://hyperimmunise.readthedocs.io/en/latest/

## Quick Start

1. Create the conda environment
```bash
conda env create -f hyperimmunise.yml
conda activate hyperimmunise
```
2. Install external dependencies

Download and install the following tools and make note of their absolute paths:

- NetNGlyc (https://services.healthtech.dtu.dk/services/NetNGlyc-1.0/)
- Jwalk from XLM-Tools (https://github.com/Topf-Lab/XLM-Tools)

3. Run the example
```bash
python main.py \
-netnglyc_loc "ABSOLUTE DIRECTORY OF netNglyc-1.0" \
-jwalk_loc "ABSOLUTE DIRECTORY OF XLM-Tools/" \
-chain "K" \
-size 1 \
-number 1 \
-priority_sites "94" \
-uncovered_sites "99-105,221-228" \
-model_glycans "N" \
-pdb_list "HG_HAhead" \
-path "ABSOLUTE DIRECTORY OF HyperImmunISE/example/test_HAhead.pdb" \
-destination "ABSOLUTE DIRECTORY OF target directory"
```
4. Expected output

The run will generate:

- `Output_0.pdb` — the designed glycosylated protein structure  
- `*_coverage_rank_searchTrajectory_scores.csv` — a log of intermediate glycosylation-site combinations evaluated during iterative pruning

Each row in `*_coverage_rank_searchTrajectory_scores.csv` corresponds to one candidate combination scored by `coverage_rank()` and reports the recursion step, predicted surface coverage, current radius cutoff, and the exact set of sites in that combination.

## How Cluster Optimization Works

HyperImmunISE starts from a list of candidate glycan sites and tries to return a small set of site combinations that satisfy two user-defined limits:

- `number` (`i`): the maximum number of final combinations to keep
- `size` (`j`): the maximum number of sites allowed in each combination

This optimization is implemented in `iterate_clusters()`, with support from `calc_overlap()`, `reduce_clusters()`, `refine_clusters()`, and `coverage_rank()`.

### Step 1. Build initial combinations

The workflow first generates candidate site combinations at an initial clash radius of `17.5` A by default. Two sites are treated as clashing if their pairwise distance is less than or equal to that radius. The initial combinations are generated by `calc_overlap()`.

At this stage, the code does not yet enforce the final `number` or `size` limits. It only enumerates combinations that can coexist under the current clash definition.

### Step 2. Keep the longest useful combinations

`refine_clusters()` simplifies the raw combination list by preferring longer combinations. It first keeps only combinations with the maximum length, then checks whether every candidate site is still represented somewhere in the reduced set.

If all sites are still represented, that reduced list is kept. If some sites disappear entirely, the minimum allowed combination length is lowered by one and the process repeats.

### Step 3. Iteratively optimize

The main optimization loop is handled by `iterate_clusters()`. At each recursion step, it checks:

- `cur_i`: the number of current combinations
- `cur_j`: the size of the largest current combination

It then applies one of four cases.

#### Case A. Too many combinations, and combinations are also too large

Condition: `cur_i > number` and `cur_j > size`

- Increase the clash radius by `1`
- Call `reduce_clusters()` to split combinations that now contain clashes
- Call `coverage_rank()` to keep only the top `number` combinations

#### Case B. Combinations are too large, but the count is acceptable

Condition: `cur_j > size` and `cur_i <= number`

- Increase the clash radius by `1`
- Call `reduce_clusters()`
- Do not rank yet

This step only fixes oversized combinations.

#### Case C. Combinations are the right size, but there are too many

Condition: `cur_i > number` and `cur_j <= size`

- Keep the current radius
- Call `coverage_rank()`
- Keep only the top `number` combinations

This step does not change geometry. It only filters by coverage.

#### Case D. Both limits are satisfied

If neither condition is violated, the current combinations are returned as final.

### How coverage ranking works

`coverage_rank()` estimates how much of the surface is covered by each candidate combination and ranks combinations by coverage fraction. For each site in a combination, nearby surface residues within the current radius are counted as covered. The top-scoring combinations are retained.

### Interpreting `*_coverage_rank_searchTrajectory_scores.csv`

The trajectory file records only recursion steps where `coverage_rank()` is called. If a round only increases the clash radius and runs `reduce_clusters()`, the recursion still advances, but no scoring rows are written for that step. As a result, iteration numbers in the CSV may be discontinuous.

Columns:

- `iteration`: recursion step at which coverage scoring was recorded
- `coverage_fraction`: fraction of all surface residues covered by at least one site in the combination at the current radius
- `covered_count`: number of surface residues counted as covered by the combination
- `total_surface_residues`: total number of surface residues considered in the coverage calculation
- `radius_A`: radius, in angstroms, used for that scoring round
- `num_sites_in_combo`: number of glycosylation sites in the candidate combination
- `sites`: semicolon-separated list of residue identifiers in the candidate combination

### Example

Suppose the user requests:

- at most `2` combinations (`number = 2`)
- at most `2` sites per combination (`size = 2`)

If the current combinations at radius `17.5` are:

- `[A, B, C]`
- `[A, D]`
- `[B, E]`

Then:

- there are `3` combinations, which exceeds `number`
- the largest combination has `3` sites, which exceeds `size`

This is Case A. The algorithm increases the radius to `18.5`, runs `reduce_clusters()`, and splits combinations that now contain clashes. If `[A, B, C]` becomes invalid because `B` and `C` now clash, the resulting valid subsets might be:

- `[A, B]`
- `[A, C]`
- `[A, D]`
- `[B, E]`

At that point, the size limit is satisfied, but there are still `4` combinations, so `coverage_rank()` is applied and only the top `2` combinations are kept, for example:

- `[A, B]`
- `[A, D]`

Now both constraints are satisfied, so the optimization stops.
